Information systems engineering is not only about writing code. It is also about reading code, testing ideas, explaining decisions, and building small reliable tools that help people work with data. A student who studies systems engineering learns to notice patterns in text, patterns in numbers, and patterns in the way users describe a problem. The same habit is useful in many areas of technology, because every larger program begins as a smaller task that must be understood carefully.

This text was prepared as sample material for a C++ program that counts words, creates a cross reference table, and finds web addresses. The topic is ordinary student work, but the text is long enough to show how associative containers can help. A map stores each different word together with a counter. A set stores the line numbers in which a word appears. A vector stores the addresses that were found while reading the file. These containers are simple, but they are very useful when a task needs searching and grouping.

At the end of the first study year many students already know variables, loops, functions, files, and several containers. They may not know every advanced library, and they do not need to know every advanced library for this work. It is enough to write clear functions, use meaningful names, and test the program with a real text file. When the program reads a line, it can split the line into smaller parts. After that it can remove punctuation from words and convert letters to lower case. Then words such as Program, program, and program. are counted as the same word.

The Vilnius university page can be written in a full form such as https://www.vu.lt/ or in a shorter form such as www.vu.lt. Another address might be https://vu.lt/. A faculty page could be mif.vu.lt, and a public learning resource could be cppreference.com. The program should find all of these forms without treating commas, brackets, or final dots as part of the address. In a real text addresses often appear inside sentences, so the cleaning step is important.

Associative containers are useful because they keep information arranged by key. When a new word is read, the program can ask the map for that word and increase its counter. If the word has not been seen before, the map creates a new entry. This makes the code short and understandable. A sequential container could also be used, but then the program would have to search through many elements again and again. For a small file the difference may be small, but for a larger file the associative container is a better choice.

The cross reference table is another example of grouping. A repeated word is more interesting when we can see where it was used. If the word student appears on lines three, seven, and ten, the output can show those line numbers near the word. A set is helpful because it stores each line number only once. If the same word appears three times on one line, the line number is still written only once. That keeps the result clean and easier to read.

Good programming style matters even in a small exercise. The program should not put all logic into one long main function. It should have separate functions for reading the text, cleaning words, checking web addresses, and writing output files. This is not complicated object oriented design, but it is already a step toward clean code. A student can later move these functions into classes if the project grows, but for this assignment simple functions are enough and easier to defend.

Testing is also part of the task. A student should run the program and inspect the produced files. The word count file should contain only words that occur more than once. The cross reference file should contain the same repeated words and their line numbers. The URL file should contain only web addresses. If an address such as https://github.com/ appears in the text, it should be written without surrounding punctuation. If an address such as github.com appears without a protocol, it should still be found.

This sample text repeats several important words on purpose. Program appears many times. Student appears many times. Text appears many times. File appears many times. Word appears many times. Container appears many times. These repeated words make it easier to see whether the counter works. The goal is not to write a perfect natural language processor. The goal is to practice string processing, maps, sets, vectors, file input, file output, and a little bit of careful thinking.

In practical work a programmer often starts from a simple requirement and then improves the solution. First the program reads the file. Then it counts words. Then it writes results. Then it adds cross reference data. Then it adds URL search. Each improvement should still leave the program understandable. When the code is prepared for a repository, the README file should explain what the program does, what input file it uses, what output files it creates, and how to run the program.

The project can be opened in Visual Studio as a console application. It can also be compiled with a C++ compiler from the command line if the compiler is available. The important part is that the source code uses standard C++ features, not platform specific code. Standard input and output files make the project easy to check. The file tekstas.txt should be near the executable when the program is run, because the program reads that file by name.

Another useful habit is to keep repository contents tidy. Source files, a README file, and a sample input file are enough for this exercise. Generated output files can be kept when they demonstrate the result, but very large temporary folders should not be committed. A small .gitignore file can ignore Visual Studio build directories such as Debug, Release, x64, and hidden local files. That way the repository stays focused on the code and the assignment.

Documentation should be short but helpful. It should mention that the task uses std::string, std::map, std::set, and std::vector. It should mention that punctuation is removed from words. It should mention that addresses may be written as https://example.com, http://example.com, www.example.com, or example.com. It should also mention that the program prints only words that appear more than once. These details help a reviewer understand the intended behavior before reading every line of code.

During defense a lecturer may ask why a map was chosen. A good answer is that the map associates a word with its count and keeps keys sorted. The lecturer may ask why a set was chosen for line numbers. A good answer is that the set avoids duplicate line numbers and keeps them sorted. The lecturer may ask why a vector was chosen for URL addresses. A good answer is that addresses can be stored in the order in which they are found. Simple answers are often the strongest answers.

The final result is a small but complete text analysis tool. It reads a text, finds repeated words, records line references, extracts web addresses, and writes clear output files. It uses associative containers where they fit naturally. It uses strings for cleaning and comparing text. It is not too large, not too abstract, and not too advanced for a first year student. That is exactly the kind of program that can show understanding without pretending to be a professional framework.

For additional examples, the text includes several more addresses: openai.com, https://learn.microsoft.com/, www.github.com, github.com, https://cplusplus.com/reference/, and docs.github.com. These addresses are here only for testing. The program should place them into the URL output file, while normal words around them should continue to be counted. If the result files show repeated words, correct line references, and all addresses, then the assignment requirements are satisfied.
